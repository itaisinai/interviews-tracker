## Configuration

### Extension Configuration (`src/auth.js`)

```javascript
{
  domain: "dev-c1s005zh8spezp0e.us.auth0.com",
  clientId: "hlI5kn4lePStXeHJohsGqyKnyoBHJtTW",
  audience: "https://interviews-tracker-api.com",
  redirectUri: chrome.identity.getRedirectURL(), // Runtime-generated by Chrome
  scope: "openid profile email offline_access"
}
```

**Important**: The redirect URI is generated by Chrome at runtime via `chrome.identity.getRedirectURL()`. The format is typically `https://<extension-id>.chromiumapp.org/` but this is NOT guaranteed. Always check the console output to get the actual URL.

### Required Auth0 Configuration

**IMPORTANT**: Before the extension works, you must:

1. Load extension in Chrome
2. Open the extension popup
3. Open DevTools console (right-click popup → Inspect)
4. Copy the redirect URI logged to console
5. Add that EXACT URL to Auth0: Applications → Your App → Allowed Callback URLs
6. Configure refresh tokens in Auth0:
   - Enable "Refresh Token Rotation"
   - Ensure API allows `offline_access` scope
   - Without this, users must re-authenticate every ~1 hour
7. Click Save Changes

## Technical Highlights

### PKCE Flow (RFC 7636)

1. Generate random `code_verifier` (43-128 chars, A-Z a-z 0-9 . _ ~ -)
2. Calculate `code_challenge` = BASE64URL(SHA256(code_verifier))
3. Redirect to Auth0 with challenge
4. Auth0 redirects back with authorization code
5. Exchange code + verifier for token
6. Auth0 validates: SHA256(provided_verifier) == stored_challenge

No client secret needed → safe for public clients like Chrome extensions.

### Token Management

## Known Limitations

1. **Single tenant**: Extension is configured for one Auth0 tenant
   - To support multiple tenants, would need configuration UI

2. **Chrome only**: Uses Chrome-specific `chrome.identity` API
   - Firefox/Edge have similar but different APIs

3. **Single email**: API validates against `ALLOWED_EMAIL` env var
   - Multi-user support would require API changes

4. **Extension ID changes**: Unpacked vs. packaged vs. published have different IDs
   - Must update Auth0 callback URL for each

5. **Token visibility**: Stored in `chrome.storage.local`
   - Chrome encrypts it, but not end-to-end encrypted
   - Users with filesystem access to Chrome profile could extract it
